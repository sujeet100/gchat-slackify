# Performance Review ŌĆö `tagger.js` / `styles.js` / `apply.js` / `config.js`

Reviewed: 2026-06-27. Scope: the restyling MV3 content script for Google Chat (a "Wiz"
framework SPA that re-renders constantly). Goal: **zero perceptible CPU/memory overhead**.
This review is read **through a performance lens** and is aligned with `CLAUDE.md`'s existing
"lightweight doctrine."

**Verdict up front:** the architecture is genuinely good ŌĆö observer-only-collects, idle-time
chunked work, WeakSet/Weak-keyed caches, no `:has()` in generated CSS, `getComputedStyle`
gated to once-per-topic. The findings below are *refinements on a sound base*, not rewrites.
The two highest-impact items (observer scope, and pruning the once-per-pass `getComputedStyle`
sweeps) close the only paths by which a busy conversation can still cost us measurable work.

## 2. What the code already does well (keep these)

- **Observer-only-collects.** The `MutationObserver` callback does *only* `pending.add(n)` + `schedule()`. No `getComputedStyle`, no `querySelectorAll`, no layout reads in the callback. This is the single most important rule and it is honored. ([MDN/best-practice: keep callbacks lightweight](https://www.dhiwise.com/blog/design-converter/mastering-the-mutation-observer-a-comprehensive-guide))
- **Idle-time, throttled, chunked.** Work runs in `requestIdleCallback` with a `timeout:1000` fallback, throttled to one scheduled pass (`if (scheduled) return`), and the topic loop respects `deadline.timeRemaining() > 3` then re-schedules ŌĆö textbook adaptive chunking. ([w3c idle-callback spec](https://w3c.github.io/requestidlecallback/), [Vercel react-best-practices rule](https://github.com/vercel-labs/agent-skills/blob/main/skills/react-best-practices/rules/js-request-idle-callback.md))
- **`getComputedStyle` is gated to once-per-topic** via `processedTopics` (WeakSet) and once-per-rail via `lightScanned`. A pass with nothing new does **zero** forced recalc. This is the correct discipline for the most expensive primitive. ([forces-layout list](https://gist.github.com/paulirish/5d52fb081b3570c81e3a))
- **No `:has()` in generated CSS.** Containers (`rail`, `stream`, `active`) are resolved in JS and tagged; CSS targets the cheap attribute selector. This avoids `:has()` re-evaluation across Chat's huge mutating DOM. ([CSS selector cost](https://developer.chrome.com/docs/performance/insights/slow-css-selector), [web.dev recalc scope](https://web.dev/articles/reduce-the-scope-and-complexity-of-style-calculations))
- **Cheap triage before expensive work.** The triage loop uses only `isConnected`, `contains`, and `closest`/`querySelectorAll(TOPIC_SEL)` ŌĆö no style/layout ŌĆö to route nodes into `topicQueue`. Expensive scans are deferred. Matches the "do the minimum, skip outside-pane nodes with a cheap `contains`" doctrine.
- **GC-friendly caches.** `WeakSet` for `lightScanned`/`processedTopics`/`datedNodes` means cached nodes don't pin memory; when Wiz drops a topic, the cache entry is collectible. No `Map`/`Set` of nodes that would leak.
- **Defensive `try/catch` everywhere** so the host app can never break, and per-selector `try/catch` in `firstMatchEl`/`allMatchEls` so a malformed/unsupported selector degrades gracefully.
- **Theme switching is an attribute flip**, not a stylesheet rebuild ŌĆö the sheet is injected once and CSS custom properties + `html[data-sf-*]` gates do the rest. Zero recompute on preference change.

## 3. Correctness risks introduced by perf optimizations

- **C1 ŌĆö `processedTopics` idempotency can leave a topic un-tagged if Chat re-populates it after first scan.** A topic `c-wiz[data-topic-id]` may be added empty (skeleton) and filled later; if our pass scans it while empty, `processedTopics.has(topic)` is now `true` and the later content never gets bubble/date tags. **Mitigation:** key the WeakSet add *after* we confirm the topic has children, or re-allow a re-scan when a mutation targets an already-processed topic that has gained children. (Low frequency, but a real "first message in a new topic isn't flattened" bug.)
- **C2 ŌĆö re-targeting the observer to `[role=main]` (P1) misses rail/topbar mutations.** If you narrow the observer you **must** add a second observer for the rail subtree, or theme/active/unread tagging stops updating when the sidebar re-renders. The fix for P1 explicitly includes the second observer ŌĆö don't ship the narrowing without it.
- **C3 ŌĆö IntersectionObserver gating (P7) changes *when* tags appear.** A topic scrolled into view tags slightly later than today's eager scan. With a generous `rootMargin` this is imperceptible, but a too-tight margin would show a flash of un-flattened grey bubble as it scrolls in. Use `rootMargin` Ōēź ~one viewport.
- **C4 ŌĆö read/write phase split (P5) must preserve the `data-slackify` skip check.** Today `if (el.hasAttribute('data-slackify')) continue;` runs inline. When splitting phases, do the skip check in the *read* phase so you don't collect already-tagged elements; otherwise the write phase could overwrite a more-specific tag (e.g. `datewrap`).
- **C5 ŌĆö `tagActiveRow` scoping to the rail (P3)** assumes the active conversation row is always inside `[data-slackify="rail"]`. It is ŌĆö but if `tagRail` hasn't resolved yet (rail not tagged on the first pass), the fallback to `document` must remain (`(rail||document)`), or active-row highlighting silently fails on first paint.

## 4. Principles (paste-ready for `CLAUDE.md`)

These extend the existing "Performance & lightweight doctrine." Each is a rule, not advice:

1. **Observe the narrowest stable subtree, not `documentElement`.** Attach the broad observer only until `[role=main]` resolves, then `disconnect()` and re-`observe()` the pane; add a separate narrow observer for the rail. A global `subtree` observer wakes us for compose typing, tooltips, and presence churn we don't care about. ([target specific containers](https://www.dhiwise.com/blog/design-converter/mastering-the-mutation-observer-a-comprehensive-guide))
2. **A pass must be region-dirty-flag driven.** During triage, set `railDirty`/`paneDirty`; only run a region's scan function when its flag is set. "Cached early-return" is not the same as "didn't run" ŌĆö a function called every pass still costs a `document.querySelector`.
3. **Always scope `querySelector(All)` to the smallest known root.** Never `document.querySelectorAll` for something that lives in the rail or one pane. Pass the resolved region node as the root. ([recalc/scan cost Ōēł elements ├Ś selectors](https://web.dev/articles/reduce-the-scope-and-complexity-of-style-calculations))
4. **Split read phases from write phases.** Within any loop that calls `getComputedStyle`/`getBoundingClientRect`, collect all reads first, then apply all `setAttribute`/style writes ŌĆö never interleave. Interleaving forces repeated style/layout recalc ("thrashing"). ([forces-layout list](https://gist.github.com/paulirish/5d52fb081b3570c81e3a), [batch reads/writes](https://www.debugbear.com/blog/forced-reflows))
5. **Prefer specific candidate selectors over `*`.** A full-subtree `querySelectorAll('*')` + `getComputedStyle` per node is the most expensive sweep; narrow to `button,[role="button"],[role="listitem"],[jsaction]` (or `div,span` only where text/bubbles can appear). ([avoid universal/complex selectors](https://web.dev/articles/reduce-the-scope-and-complexity-of-style-calculations))
6. **Chunk every unbounded sweep against the idle `deadline`, including rail re-scans.** It is not enough to budget the *number* of items; a single large sweep (rail swap, giant topic) must yield via `deadline.timeRemaining()` and `schedule()`. ([adaptive chunking](https://w3c.github.io/requestidlecallback/))
7. **Scan only what the user can see ŌĆö gate per-topic work behind `IntersectionObserver`** (`rootMargin` Ōēź one viewport), so bulk-loaded back-scroll history costs nothing until scrolled to. `unobserve` after the one-time scan. ([IntersectionObserver processes only what matters](https://medium.com/codex/intersection-observer-vs-mutation-observer-71fd9b8b757d))
8. **Re-scan idempotency must tolerate late-filled containers.** Add a node to the "processed" WeakSet only after confirming it has content, or permit one re-scan when a processed topic gains children ŌĆö otherwise skeleton-then-fill renders escape tagging.
9. **Never disconnect without re-attaching; never observe without an unobserve plan.** Each new observer needs a documented teardown (`disconnect`/`unobserve`) so a long-lived SPA session doesn't accumulate observer node sets. Keep node caches `WeakSet`/`WeakMap`-keyed.
10. **Do not apply `content-visibility` / `contain` / `will-change` to host (Google-owned) nodes.** We don't own Chat's rendering strategy; forcing containment can fight Wiz's virtualization and break find-in-page/anchoring. These are deliberate non-goals. ([content-visibility caveats](https://web.dev/articles/content-visibility))
11. **Keep CSS injected once and gate with attributes.** Generate the sheet from config, inject one `<style>` at `document_start`, switch features/themes by flipping `html[data-sf-*]` ŌĆö never rebuild the sheet on preference change. (Already done; keep it.) ([content_scripts CSS injects before DOM](https://developer.chrome.com/docs/extensions/develop/concepts/content-scripts))
